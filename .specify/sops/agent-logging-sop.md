## 1. Purpose

All agents MUST log their work to Airtable for the following reasons:

- **Accountability**: Track all agent activity for project management and performance reviews
- **Resource Planning**: Understand time allocation across projects for capacity planning
- **Token Budgeting**: Monitor AI resource consumption to optimize costs
- **Project Health**: Feed data into health score calculations for each project
- **Audit Trail**: Maintain complete record of all work performed by the digital team
- **Billing Accuracy**: Support accurate client billing where applicable

**Reference**: Constitution Section IX.III - Time & Task Tracking (MANDATORY - NO EXCEPTIONS)

---

## 2. When to Log

### REQUIRED Logging Events

| Event | Action Required |
|-------|-----------------|
| After EVERY task completion | Log Time Entry |
| Before ending ANY session | Log Time Entry (non-negotiable) |
| Multi-step work completed | Log Task Record + Time Entry |
| Work >5 minutes | Log Task Record + Time Entry |
| Deliverable produced | Log Task Record + Time Entry |

### Critical Rule

**NO agent session ends without Airtable logging.** This is enforced without exception per Constitution Section VII.VI.

### 3.2 Task Record (Conditional)

**Table**: Tasks
**Table ID**: `tbliXF3imV0uFxJSB`

Create a Task record when work meets ANY of these criteria:

- Takes more than 5 minutes
- Produces a deliverable (document, code, report, etc.)
- Involves multiple steps
- Should be tracked for project health scoring
- Represents a milestone or checkpoint

**Task Record Fields** (check Airtable schema for current fields):
- Task Name
- Agent (linked)
- Project (linked)
- Status
- Description
- Deliverable Type
- Time Entries (linked)

Request Tab to log on your behalf:

### 4.3 Via Direct API (Fallback)

If MCP is unavailable, use Airtable REST API with proper authentication.

## 5. Token Tracking

### How to Get Token Counts

Agents should obtain token usage from their session reports:

1. **Claude Code Sessions**: Check the token usage displayed in the interface or API response
2. **API Responses**: Extract from `usage` object in API response:
   - `input_tokens`: Tokens sent to the model
   - `output_tokens`: Tokens generated by the model
   - **Total**: `input_tokens + output_tokens`

### Token Reporting Requirements

- Report TOTAL tokens (input + output combined)
- If exact count unavailable, estimate based on session length
- Round to nearest whole number
- Never leave Tokens Used blank - 0 is not acceptable for completed work

### Estimation Guidelines (if metrics unavailable)

| Task Type | Estimated Tokens |
|-----------|------------------|
| Quick lookup/answer | 1,000-3,000 |
| Simple file edit | 3,000-8,000 |
| Multi-file analysis | 8,000-20,000 |
| Complex implementation | 20,000-50,000 |
| Large refactor/project | 50,000+ |

## 6. Common Mistakes

### NEVER Do These

| Mistake | Impact | Prevention |
|---------|--------|------------|
| Forgetting to log tokens | Incomplete resource tracking | Always include token count, estimate if needed |
| Not logging before session end | Lost work attribution | Make logging the LAST action every session |
| Skipping task records for significant work | Missing project health data | Ask: "Did I produce something? >5 min?" |
| Logging to wrong project | Incorrect billing/allocation | Verify project name before logging |
| Vague descriptions | Audit trail useless | Include: what, why, outcome |
| Rounding hours to 0 | Underreporting effort | Use 0.02 minimum (1 minute) |
| Batching multiple days | Inaccurate daily tracking | Log same-day, every day |

### Red Flags Drew Monitors

- Agent session without corresponding time entry
- Time entry with 0 or missing tokens
- Significant work without task record
- Large gaps in logging timeline
- Mismatched project assignments

## 7. Verification

### Drew/PM Verification Process

Drew performs the following compliance checks:

#### Daily
- Review time entries from previous day
- Flag any sessions without entries
- Check for missing token counts

#### Weekly (Friday)
- Cross-reference agent sessions with time entries
- Verify task records exist for major deliverables
- Calculate health scores per project
- Identify logging compliance issues

#### Compliance Metrics

| Metric | Target | Escalation |
|--------|--------|------------|
| Time Entry Rate | 100% of sessions | Missing entry = immediate follow-up |
| Token Logging Rate | 100% of entries | Missing tokens = agent notification |
| Task Record Rate | 100% of deliverables | Missing tasks = weekly review item |

### Self-Verification Checklist (Before Ending Session)

- [ ] Time entry created in Airtable
- [ ] Entry Date is today's date
- [ ] Correct Agent linked
- [ ] Correct Project linked
- [ ] Hours are accurate (decimal format)
- [ ] Description is clear and specific
- [ ] Tokens Used is populated
- [ ] Task record created (if applicable)
- [ ] Learnings.md updated (if new patterns discovered)
